# Realtime Transcription — Delivery

Owns the right side of the pipeline: everything after core pipeline confirms a segment.

## Scope

```
Core pipeline confirms segment
    ↓
SegmentPublisher → Redis XADD
    ↓
Transcription Collector (stream consumer)
    ↓
Redis Hash (mutable store)     →  Redis PUBLISH → API Gateway WS → Dashboard
    ↓
Background DB Writer (10s cycle, 30s immutability)
    ↓
Postgres
    ↓
REST GET /transcripts/{platform}/{native_id}
    ↓
Dashboard (page load/refresh)
```

## What "correct" means

Given N segments from the core pipeline:

1. **WS delivers all N** to any subscriber — no drops, no phantom segments
2. **REST returns all N** after persistence — no drops from dedup
3. **Dashboard renders all N** — no duplicates, no disappearing, correct order
4. **Ordering** — segments sorted by start_time, cross-speaker interleaving preserved
5. **Latency** — WS delivers within 1s of publish, REST within 30s+10s of publish

## Known bugs

### WS: authorization_service_error on replay meetings
- Replay creates meeting directly in DB (bypasses bot-manager)
- Collector's `/ws/authorize-subscribe` endpoint can't find the meeting
  for the subscribing user because the meeting wasn't created through
  the normal API flow
- Fix: either create meetings through the API, or fix the auth endpoint

### REST: 15/43 segments lost in dedup
- Collector endpoint dedup drops segments despite speaker check fix
- Suspected: the dedup compares only adjacent segments (last vs current)
  but per-segment splitting creates non-adjacent overlaps
- Need to trace exactly which segments are dropped and why

### Dashboard: draft→confirmed text shrinking
- Draft shows full accumulated text
- Confirmed is a per-segment split (shorter text, same start time)
- Dashboard replaces draft with shorter confirmed → text "disappears"
- Root cause: per-segment confirm was too eager (confirmThreshold=1)
- Partially fixed: now requires confirmThreshold=2 matches
