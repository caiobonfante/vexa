1. Meetings — bots join Google Meet / Teams / Zoom, transcribe live
↓

from real conversations, not manually authored

Agent Runtime

Like K8s — but for agents


  • Sandboxed — every agent works in an isolated container
  • Git-controlled — workspace is version-controlled, persistent
  • Can't escape — agents do anything inside the sandbox, nothing outside
  • Self-scaling — zero idle cost, containers come and go
  • Ephemeral — spin up to do a thing, then die

Three Container Types


ContainerRAMPurpose
Browser ~1.5 GB Meeting bots — joins calls, VNC + CDP
Agent ~200 MB Git-controlled workspace — summarize, extract, generate CALM
Worker ~50 MB Fire-and-forget — webhooks, notifications

Browser doesn't carry an agent. Agent doesn't carry Chromium.
They connect over the network when needed, then die independently.

Scheduler

No human in the loop


9:00 AM weekday
→ Browser container spawns, joins standup
→ Transcribes the meeting live
→ Meeting ends → agent container wakes up
→ Extracts decisions, updates CALM specs
→ Flags architecture conflicts → posts to Slack
→ All containers die. Zero cost until tomorrow.

Cron + event triggers + container chaining (on_success / on_failure)

Why CALM?


Meetings produce decisions.

Decisions shape architecture.

Architecture should be code.


Standup: "we're splitting the auth service"
→ agent extracts decision
→ updates CALM spec
→ validates against existing architecture
→ flags conflicts
→ posts back to team

Architecture-as-code becomes living — updated from real conversations.
